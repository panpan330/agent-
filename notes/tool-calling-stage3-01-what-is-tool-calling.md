# 阶段 3 第 1 节：Tool Calling 是什么

## 本节目标

这一节先不写业务代码。

阶段 3 要做的事情，是让 AI 服务逐步具备“调用工具”的能力。后面我们会让 Python AI 服务调用一个 Java mock 业务服务，再让它完成查订单、查物流、创建工单这类动作。

在真正写代码前，必须先把一个核心概念讲透：

```text
Tool Calling 不是模型自己去执行代码。
Tool Calling 是模型告诉后端：我想调用哪个工具，以及我准备传什么参数。
真正执行工具的是我们的程序。
```

你要先记住这句话，后面所有实现都围绕它展开。

## 先用一句人话理解

普通聊天像这样：

```text
用户：帮我查一下订单 A1001 的物流。
模型：你可以登录订单系统查看物流信息。
```

这只是“回答问题”。

Tool Calling 像这样：

```text
用户：帮我查一下订单 A1001 的物流。
模型：我需要调用 query_order_logistics 工具，参数是 {"order_id": "A1001"}。
后端：收到，我去调用真实业务接口。
Java 业务服务：返回物流状态。
后端：把物流状态再交给模型。
模型：订单 A1001 当前已经发往上海转运中心，预计明天送达。
```

这就不只是“生成文字”，而是让 AI 参与了业务流程。

## 为什么需要 Tool Calling

大模型本身擅长的是理解和生成文本：

- 理解用户自然语言。
- 判断用户意图。
- 把模糊问题整理成结构化参数。
- 根据上下文组织回答。
- 对工具返回的结果做总结。

但大模型本身不应该直接负责：

- 查询数据库。
- 修改订单状态。
- 创建退款单。
- 调用内部 Java 接口。
- 判断用户有没有权限。
- 记录审计日志。
- 控制幂等性。

原因很简单：这些是业务系统的责任，不是模型的责任。

如果让模型“直接操作业务系统”，会有几个严重问题：

- 权限不可控：模型可能尝试执行用户无权执行的操作。
- 参数不可控：模型生成的参数可能缺字段、类型错、范围错。
- 行为不可审计：如果没有后端统一记录，就很难知道谁做了什么。
- 结果不可重复：模型输出有概率性，同一句话不一定每次都一样。
- 错误不好处理：Java API 超时、404、500、限流，都需要工程化处理。
- 敏感操作危险：退款、删除、改状态这类操作不能让模型一句话就执行。

所以正确边界是：

```text
模型负责“理解和决策建议”。
后端负责“校验、权限、执行、日志、错误处理”。
```

## Tool 在这里是什么意思

这里的 tool 可以理解成“后端暴露给模型看的能力说明书”。

一个工具一般包含三部分：

| 部分 | 含义 | 例子 |
| --- | --- | --- |
| name | 工具名称 | `query_order` |
| description | 工具说明 | 查询订单详情，适合用户询问订单状态时使用 |
| parameters | 参数结构 | `order_id` 必填，类型是字符串 |

也就是说，工具不是随便写一句“你可以查订单”。

工具要写成明确的契约：

```json
{
  "name": "query_order",
  "description": "查询订单详情，适合用户询问订单状态、支付状态或发货状态时使用。",
  "parameters": {
    "type": "object",
    "properties": {
      "order_id": {
        "type": "string",
        "description": "订单编号，例如 A1001"
      }
    },
    "required": ["order_id"]
  }
}
```

这段 JSON 的重点不是格式本身，而是思想：

```text
我们把后端能力用机器能读懂的方式描述给模型。
模型根据这些描述决定是否需要调用工具。
```

## Tool Calling 的完整流程

以后你看到 Tool Calling，要先在脑子里画出这条链路：

```text
用户问题
-> 模型判断是否需要工具
-> 模型返回 tool call 和 arguments
-> 后端校验工具名和参数
-> 后端检查权限、限流、幂等性
-> 后端执行真实工具或调用 Java API
-> 后端拿到工具结果
-> 后端把工具结果返回给模型
-> 模型总结成自然语言回复给用户
```

用智能工单 Agent 举例：

```text
用户：订单 A1001 三天没发货了，帮我催一下。

模型第一次响应：
需要调用 query_order 工具，参数 {"order_id": "A1001"}。

Python AI 服务：
校验参数，调用 Java 订单服务。

Java 订单服务：
返回订单状态：已付款，未发货，仓库处理中。

Python AI 服务：
把工具结果发回模型。

模型第二次响应：
订单 A1001 已付款，目前仓库正在处理中。是否需要我帮你创建催发货工单？

用户：需要。

模型：
需要调用 create_ticket 工具，参数 {"order_id":"A1001","type":"logistics_delay","summary":"用户反馈订单三天未发货"}。

Python AI 服务：
先检查是否需要用户确认、检查权限、校验参数，再调用 Java 工单服务。
```

注意这里有一个关键点：

```text
模型不是直接创建工单。
模型只是提出“要调用 create_ticket”。
后端决定是否真的执行。
```

## 普通聊天、结构化输出、Tool Calling 的区别

我们前面阶段 2 已经学过 `/chat` 和 `/extract-ticket`，现在要把三者区分清楚。

| 能力 | 目标 | 模型返回什么 | 后端是否执行动作 |
| --- | --- | --- | --- |
| 普通聊天 | 回答用户问题 | 自然语言文本 | 不执行业务动作 |
| 结构化输出 | 抽取字段或生成固定格式数据 | JSON 数据 | 通常不直接执行动作 |
| Tool Calling | 决定调用哪个工具并给出参数 | 工具名 + 参数 | 后端可以执行业务动作 |

举例：

普通聊天：

```text
用户：我想退款。
模型：请提供订单号，我可以帮你说明退款流程。
```

结构化输出：

```json
{
  "intent": "refund",
  "order_id": "A1001",
  "summary": "用户希望退款",
  "urgency": "normal",
  "need_human_review": true
}
```

Tool Calling：

```json
{
  "tool_name": "query_refund_policy",
  "arguments": {
    "order_id": "A1001"
  }
}
```

更进一步，后端可能真的调用：

```text
GET /orders/A1001/refund-policy
```

所以你可以这样理解：

```text
结构化输出重点是“把话变成数据”。
Tool Calling 重点是“把话变成可执行的业务调用意图”。
```

## Tool Calling 和 JSON Schema 的关系

工具参数需要被清楚描述，否则模型不知道应该传什么。

常见做法是用 JSON Schema 描述参数结构。

JSON Schema 的作用是：

```text
规定一个 JSON 对象应该长什么样。
```

例如查询订单需要 `order_id`：

```json
{
  "type": "object",
  "properties": {
    "order_id": {
      "type": "string"
    }
  },
  "required": ["order_id"]
}
```

这表示：

- 参数整体是一个对象。
- 对象里可以有 `order_id`。
- `order_id` 必须是字符串。
- `order_id` 是必填字段。

但你要注意：

```text
JSON Schema 是参数说明。
后端校验仍然必须做。
```

不能因为模型看过 schema，就相信它永远传对参数。

后端仍然要用 Pydantic 或 Java DTO 校验：

- 字段是否存在。
- 类型是否正确。
- 字符串是否为空。
- 枚举值是否合法。
- 数字是否在允许范围内。
- 用户是否有权限访问这个资源。

## 模型为什么不能自己直接调用 Java API

很多初学者会误以为：

```text
既然叫 Tool Calling，那是不是模型直接请求 Java 接口？
```

不是。

一般情况下，模型只返回一个结构化的 tool call。这个 tool call 会被你的 Python AI 服务收到。然后 Python AI 服务再去调用 Java API。

也就是说，后面的架构大概会变成：

```text
用户
-> Python FastAPI AI 服务
-> LLM
-> Python FastAPI AI 服务解析 tool call
-> Java 业务服务
-> Python FastAPI AI 服务
-> LLM
-> 用户
```

为什么要这样绕一层？

因为 Python AI 服务要承担“AI 编排层”的责任：

- 管理模型调用。
- 管理工具列表。
- 校验模型返回的工具参数。
- 调用 Java 服务。
- 处理 Java 服务错误。
- 记录 trace_id。
- 控制日志。
- 做权限和确认逻辑。
- 决定是否把工具结果再次交给模型总结。

Java 服务继续承担“业务系统层”的责任：

- 订单数据。
- 工单数据。
- 用户权限。
- 业务规则。
- 数据库事务。
- 审计记录。

这就是我们这条学习路线里“Java 后端 + Python AI 服务”的分工。

## Tool Calling 的两个阶段

实际开发时，Tool Calling 通常不是一次模型调用就结束，而是至少分成两段。

第一段：让模型决定要不要调用工具。

```text
输入：用户问题 + 工具说明
输出：模型返回 tool call
```

第二段：后端执行工具后，把结果交回模型总结。

```text
输入：用户问题 + 工具结果
输出：模型给用户的最终回答
```

所以完整交互经常是：

```text
模型调用 1：理解用户问题，决定工具。
真实工具调用：后端执行。
模型调用 2：根据工具结果组织回答。
```

这也是为什么 Tool Calling 比普通聊天更复杂：

- 调用次数可能更多。
- 错误分支更多。
- 日志链路更长。
- 测试要覆盖模型、工具、工具结果、最终回答。

## 后端拿到 tool call 后必须做什么

以后写代码时，不能一拿到模型的 tool call 就直接执行。

至少要经过这些步骤：

| 步骤 | 为什么需要 |
| --- | --- |
| 校验工具名 | 防止模型返回不存在或不允许的工具 |
| 校验参数结构 | 防止缺字段、类型错、格式错 |
| 检查权限 | 防止用户查询或操作不属于自己的资源 |
| 判断是否敏感操作 | 创建、修改、退款、删除通常需要用户确认 |
| 做幂等控制 | 防止同一个请求重复创建多张工单 |
| 设置超时 | 防止 Java API 卡死拖垮 AI 服务 |
| 处理错误 | 404、500、超时、限流都要转成可理解的错误 |
| 记录日志 | 记录工具名、耗时、状态、trace_id，不记录敏感内容 |

这就是工程化 Tool Calling 的核心。

## 什么工具适合交给模型决定

适合的工具：

- 查询订单状态。
- 查询物流状态。
- 查询退款政策。
- 查询知识库。
- 创建待确认的工单草稿。
- 对用户问题分类。
- 生成候选解决方案。

需要谨慎的工具：

- 创建正式工单。
- 修改订单备注。
- 提交退款申请。
- 修改用户信息。
- 触发短信或邮件通知。

高风险工具：

- 删除数据。
- 直接退款。
- 修改支付状态。
- 修改权限。
- 批量操作。

高风险工具不是绝对不能做，而是必须有更严格的规则：

- 用户明确确认。
- 后端权限校验。
- 参数二次校验。
- 幂等键。
- 审计日志。
- 必要时人工审核。

## 以后在项目里怎么落地

阶段 3 后面的学习会按这条路线走：

```text
先用 fake tool 模拟工具调用
-> 再用 FastAPI 写 Java mock 业务服务
-> Python AI 服务调用 Java mock API
-> 让模型决定是否调用工具
-> 工具结果再交给模型总结
-> 加入权限、确认、日志、trace_id、测试
-> 再引入 LangChain 的 Tool 抽象
```

为什么不是一上来就 LangChain？

因为如果你还不理解 Tool Calling 的原始流程，一上来用 LangChain 只会记住框架写法，不知道底层发生了什么。

我们这一阶段的教学顺序会先理解原理，再用框架提高效率。

## 一个完整例子

假设后端提供两个工具：

```text
query_order(order_id)
create_ticket(order_id, category, summary)
```

用户说：

```text
订单 A1001 一直没发货，帮我处理一下。
```

模型应该先调用查询工具，而不是直接创建工单：

```json
{
  "tool_name": "query_order",
  "arguments": {
    "order_id": "A1001"
  }
}
```

后端调用 Java 订单服务，得到：

```json
{
  "order_id": "A1001",
  "paid": true,
  "shipping_status": "not_shipped",
  "days_since_paid": 3
}
```

模型再根据结果回复：

```text
我查到订单 A1001 已付款 3 天，但还没有发货。这个情况可以创建催发货工单。是否需要我帮你创建？
```

用户确认：

```text
需要。
```

模型再提出创建工单：

```json
{
  "tool_name": "create_ticket",
  "arguments": {
    "order_id": "A1001",
    "category": "shipping_delay",
    "summary": "用户反馈订单已付款 3 天仍未发货，需要催发货。"
  }
}
```

后端执行前要检查：

- 当前用户是否有权操作订单 A1001。
- 用户是否已经明确确认创建工单。
- 是否已经为同一个订单创建过同类未关闭工单。
- 参数是否符合 `create_ticket` 的 schema。
- Java 工单服务是否调用成功。

最后再回复用户：

```text
已为订单 A1001 创建催发货工单，工单号是 T202607100001。
```

## 常见误解

误解 1：Tool Calling 就是模型会写代码并执行。

正确理解：

```text
模型只是返回工具调用请求，执行者是后端。
```

误解 2：Tool Calling 就不需要写接口了。

正确理解：

```text
工具背后仍然要有真实函数、HTTP API、数据库查询或业务服务。
```

误解 3：模型给了参数，后端就可以信任。

正确理解：

```text
模型输出永远要校验。工具参数尤其要校验。
```

误解 4：Tool Calling 和结构化输出一样。

正确理解：

```text
结构化输出偏数据抽取；Tool Calling 偏动作决策。
```

误解 5：LangChain 才能做 Tool Calling。

正确理解：

```text
不用 LangChain 也能做 Tool Calling。LangChain 只是后面帮我们封装工具、模型和流程。
```

## 本节你必须真正理解的句子

如果别人问你 Tool Calling 是什么，你应该能这样回答：

```text
Tool Calling 是一种让大模型参与业务系统调用的机制。
开发者先把可用工具的名称、说明和参数结构提供给模型。
模型根据用户问题判断是否需要调用工具，并返回工具名和参数。
但模型不直接执行工具，真正执行的是后端程序。
后端要校验参数、检查权限、调用真实业务接口、处理错误、记录日志，
再把工具结果交给模型生成最终回答。
```

这段话能讲明白，才算这节过关。

### 练习 3：说出后端执行前要做的检查

用户说：

```text
帮我给订单 A1001 创建投诉工单。
```

模型返回：

```json
{
  "tool_name": "create_ticket",
  "arguments": {
    "order_id": "A1001",
    "category": "complaint",
    "summary": "用户要求投诉订单 A1001。"
  }
}
```

后端直接调用 Java API 创建工单可以吗？为什么？

参考答案：

不可以直接执行。至少要检查：

- `create_ticket` 是否是允许调用的工具。
- 参数是否符合 schema，例如 `order_id`、`category`、`summary` 是否存在且类型正确。
- 当前用户是否有权操作订单 A1001。
- 这个操作是否需要用户明确确认。
- 是否已经存在同类未关闭工单，避免重复创建。
- Java 工单服务调用是否设置超时。
- 工具调用日志是否记录 trace_id、工具名、耗时和结果状态。

## 自测问题

### 1. Tool Calling 的核心边界是什么？

答案：

模型负责判断要调用哪个工具、生成参数；后端负责校验、权限、执行、错误处理和日志。模型不是执行者，后端才是执行者。

### 2. 为什么工具参数要用 schema 描述？

答案：

因为模型需要知道工具需要哪些参数、参数是什么类型、哪些字段必填。schema 是工具参数的机器可读说明书。但 schema 不能替代后端校验。

### 3. 结构化输出和 Tool Calling 最大区别是什么？

答案：

结构化输出主要是让模型按固定格式返回数据；Tool Calling 是让模型提出要调用某个工具并给出参数，后端可能因此执行真实业务动作。

### 4. Tool Calling 为什么通常需要两次模型调用？

答案：

第一次模型调用用于判断是否需要工具并生成工具参数；后端执行工具后，第二次模型调用根据工具结果生成最终自然语言回复。

### 5. 为什么创建工单这类操作不能让模型直接执行？

答案：

因为创建工单会改变业务系统状态，必须由后端做权限校验、用户确认、参数校验、幂等控制和审计日志。模型只能提出调用意图，不能越过业务系统规则。

## 本节小结

本节只需要把概念打牢：

- Tool Calling 是“模型提出工具调用，后端执行工具”。
- 工具是后端能力的契约，包含名称、说明和参数 schema。
- 模型返回 tool call，不代表一定执行，后端有最终决定权。
- Tool Calling 比普通聊天更接近真实业务系统，所以必须重视校验、权限、确认、幂等、日志和错误处理。
- 后面我们会先用 fake tool 练习，再把工具调用接到 Java mock API 上。
