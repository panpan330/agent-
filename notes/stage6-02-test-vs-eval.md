# 阶段 6 第 2 节：什么是 eval：测试和评测的区别

## 本节定位

上一节我们讲了一个总原则：

AI 应用不能只靠“我问几句，感觉还行”来判断质量。

这一节要继续往下拆：

既然不能只靠感觉，那到底应该靠什么？

你以前已经接触过 `pytest`。在 Python 基础、FastAPI、LLM API、Tool Calling、RAG、LangGraph 这些阶段里，我们一直在写测试。

比如：

- `/health` 是否返回 200。
- Pydantic 请求模型是否能校验字段。
- fake LLM 是否按预期返回。
- fake RAG 是否能返回指定答案。
- 工单字段缺失时是否进入追问节点。
- 用户确认后是否调用 Java mock 创建工单。

这些都叫测试。

现在阶段 6 又开始讲 eval。

你很自然会有一个问题：

测试和 eval 到底是不是一个东西？

如果项目里已经有 pytest 了，为什么还要 eval？

如果 eval 能评估 AI 效果，那 pytest 是不是就不重要了？

本节就是专门解决这个问题。

先给结论：

测试和 eval 有重叠，但不是同一个东西。

测试更偏向：

这段确定性的代码有没有按规则运行。

eval 更偏向：

这个 AI 应用在一批有代表性的任务上表现得好不好。

测试通常问：

有没有错？

eval 通常问：

好到什么程度？

测试通常是硬门槛。

eval 通常是质量度量。

测试适合判断：

- 函数返回值是否等于预期。
- API 状态码是否正确。
- JSON 字段是否存在。
- 异常是否被映射成统一错误码。
- 某个明确路线是否被选择。

eval 适合判断：

- 回答是否有帮助。
- 回答是否忠于知识库。
- 检索结果是否足够相关。
- 模型是否选对了工具。
- Agent 是否走了合理路线。
- 字段提取在真实表达里是否稳定。
- prompt 或模型升级后效果是否变好。

你以后做 AI 项目，不能只会说“我写了测试”，也不能只会说“我跑了 eval”。

更专业的表达应该是：

我用测试保证代码契约和关键流程不坏，用 eval 衡量模型行为和整体效果是否达到业务标准。

## 本节学习目标

学完本节后，你要能说清楚：

1. 什么是 test。

   答案：test 是对某个确定行为做验证，通常有明确输入、明确操作、明确断言，用来判断代码是否符合预期。

2. 什么是 eval。

   答案：eval 是用一批固定样本和评测标准，系统化衡量 AI 应用质量的方法。

3. 测试和 eval 的核心区别是什么。

   答案：测试更偏确定性 correctness，eval 更偏 AI 行为质量和效果度量；测试常用硬性 pass/fail，eval 可以有分数、等级、相对比较和坏例分析。

4. 为什么 AI 项目只写 pytest 不够。

   答案：pytest 能保证代码流程没坏，但很难直接衡量回答是否有帮助、检索是否相关、模型选择是否更优、prompt 改动是否让整体体验变好。

5. 为什么 eval 不能替代 pytest。

   答案：eval 衡量质量，但不能替代 API 契约、异常处理、字段校验、权限边界、写操作确认等确定性代码测试。

6. 当前智能工单 Agent 里哪些适合测试。

   答案：Pydantic schema、节点函数、路由规则、错误映射、fake 依赖、API 状态码、用户确认边界、工具调用参数校验等。

7. 当前智能工单 Agent 里哪些适合 eval。

   答案：意图识别质量、RAG 回答质量、字段提取覆盖率、工具选择准确率、Agent 路线合理性、拒答是否合适、模型/prompt 升级前后效果对比等。

8. 为什么 eval 也可以用 pytest 来跑。

   答案：pytest 是运行和断言工具，eval 是评测思想和数据结构；可以用 pytest 参数化运行 eval case，也可以后续用 LangSmith 这类平台管理数据集和实验。

9. 为什么阶段 6 先讲 test vs eval。

   答案：如果不先分清边界，后面写评测集、评测脚本和报告时，就容易把“代码测试”和“AI 效果评测”混成一团。

## 一、基础知识铺垫

### 1. 普通软件里的 test 是什么

test，中文通常叫测试。

它的核心目的不是“证明代码完美”，而是：

用可重复的方式验证某个行为是否符合预期。

比如你写了一个函数：

```python
def add(a: int, b: int) -> int:
    return a + b
```

测试可以写成：

```python
def test_add():
    assert add(1, 2) == 3
```

这段测试做了三件事：

1. 给一个明确输入：`1` 和 `2`。
2. 执行明确行为：调用 `add(1, 2)`。
3. 判断明确结果：必须等于 `3`。

如果结果是 `3`，测试通过。

如果结果是 `4`，测试失败。

这里没有模糊空间。

它不是问：

这个函数好不好？

它问的是：

这个函数有没有按约定返回结果？

所以测试非常适合检查确定性的代码行为。

确定性指的是：

同样输入，在同样条件下，应该得到同样输出。

普通软件里大量逻辑都属于确定性逻辑：

- 参数校验。
- 字段转换。
- 路由分支。
- HTTP 状态码。
- 数据库查询条件。
- 异常映射。
- 权限判断。
- 幂等键校验。
- 日志字段生成。

这些东西适合用测试守住。

### 2. pytest 官方说的测试结构

pytest 官方文档讲测试时，会把一个测试拆成几步：

- Arrange。
- Act。
- Assert。
- Cleanup。

翻译成容易理解的话：

Arrange：准备。

比如准备输入、fake client、测试数据、配置项。

Act：执行。

比如调用函数、请求 API、运行图流程。

Assert：断言。

比如判断返回值、状态码、字段、异常、日志。

Cleanup：清理。

比如清掉临时文件、关闭连接、恢复依赖。

我们项目里一直在用这个思路，只是你可能没有刻意把它拆出来。

比如测试工单字段缺失进入追问：

Arrange：

准备一条缺少手机号或问题描述的用户输入。

Act：

运行智能工单 Agent 图。

Assert：

断言结果里 `missing_ticket_fields` 有对应字段，并且路线进入追问节点。

Cleanup：

如果用了临时依赖、临时状态或覆盖配置，测试结束后恢复。

这就是典型测试。

测试的强项是：

能把“应该怎样”写成非常明确的断言。

### 3. 什么是 assert

`assert` 可以理解成：

我要求这件事必须成立。

比如：

```python
assert response.status_code == 200
```

意思是：

响应状态码必须是 200。

如果不是 200，测试失败。

再比如：

```python
assert result["intent"] == "policy_question"
```

意思是：

意图必须是 `policy_question`。

如果模型或规则返回了 `create_ticket`，测试失败。

assert 最适合表达硬规则。

硬规则指的是：

不应该商量、不应该模糊、不应该看情况。

比如：

- API key 不能写死在代码里。
- 缺少必填字段必须返回 422。
- 查询订单工具参数必须有 `order_id`。
- 创建工单前必须有用户确认。
- Java 服务超时要映射成固定错误码。
- 敏感写操作不能让模型直接执行。

这些都应该用测试守住。

如果这些都靠 eval 去评估，就会把简单问题复杂化。

### 4. 测试为什么重要

很多初学者会觉得：

我手动运行一下，没报错就行了。

这个想法在学习阶段可以理解，但在工程里不够。

因为项目一旦变大，会出现几个问题：

1. 你不可能每次改代码后手动点完所有接口。

2. 你不可能记住所有历史功能。

3. 一个新功能可能无意中破坏旧功能。

4. 没有测试，别人接手项目时不敢改。

5. 没有测试，代码重构风险很高。

测试的价值是：

把重要行为固化下来。

以后每次改动，都可以快速确认：

以前承诺过的行为还在不在。

所以测试不是为了写给老师看。

测试是项目的安全网。

尤其是我们这种长期学习项目：

每一阶段都会继续叠新能力。

如果没有测试，阶段 6 改评测、真实模型、checkpoint、tracing、Docker Compose 时，很容易把阶段 3、阶段 4、阶段 5 的旧能力改坏。

### 5. AI 应用为什么让 test 变得不够

现在问题来了。

既然测试这么重要，为什么还要 eval？

原因是：

AI 应用里有很多东西不是传统确定性逻辑。

比如你问模型：

“我订单已经付款三天了还没发货，能帮我看一下吗？”

模型可能返回：

```text
我可以帮你查询订单状态，请提供订单号。
```

也可能返回：

```text
请把订单号发给我，我帮你确认物流和发货情况。
```

也可能返回：

```text
你可以联系客服处理。
```

这些回答不一定逐字相同。

但是前两个可能都算合理。

第三个可能就比较弱，因为它没有推动流程。

传统测试喜欢判断：

实际值是否等于预期值。

但 AI 输出很多时候不能要求逐字相等。

你不能简单写：

```python
assert answer == "请提供订单号。"
```

因为合理回答可能有很多种表达方式。

这就是 AI 应用里 eval 的重要原因。

eval 要处理的问题是：

输出不完全固定，但仍然要判断质量。

### 6. 什么是 eval

eval 是 evaluation 的简写。

中文可以叫评测。

在 AI 应用工程里，eval 的核心是：

用一批固定样本和明确标准，衡量 AI 应用的效果。

这里有三个关键词：

固定样本。

明确标准。

衡量效果。

固定样本指的是：

不是今天随便问三句，明天又随便问五句。

而是维护一批可以反复运行的 case。

比如：

- 退款规则问题。
- 账号安全问题。
- 订单查询问题。
- 明确投诉问题。
- 缺字段工单问题。
- prompt injection 问题。
- 无资料问题。

明确标准指的是：

每条 case 不能只写“感觉不错”。

要写清楚希望它做到什么。

比如：

- 预期 intent 是 `policy_question`。
- 预期 route 是 `rag_answer`。
- 预期必须引用知识库。
- 预期不能编造退款期限。
- 预期缺少订单号时必须追问。
- 预期不能调用写操作工具。

衡量效果指的是：

跑完之后要能得到结果。

比如：

- 20 条样本里 18 条通过。
- 意图识别准确率 90%。
- 工单字段提取 15 条里 13 条正确。
- RAG 无资料拒答样本全部通过。
- 工具调用参数错误集中出现在订单号表达上。

这就从“感觉”变成了“证据”。

### 7. eval 不是单纯让大模型打分

很多人一听 eval，就以为：

找另一个大模型给回答打分。

这只是 eval 的一种方式，不是全部。

eval 可以很简单。

比如检查 intent：

```text
actual_intent == expected_intent
```

这就是 eval。

检查 route：

```text
actual_route == expected_route
```

这也是 eval。

检查字段：

```text
actual_fields.order_id == expected_fields.order_id
```

这也是 eval。

检查回答有没有引用来源：

```text
len(citations) > 0
```

这也是 eval。

检查回答是否包含禁用内容：

```text
not contains_private_key(answer)
```

这也是 eval。

只有当判断标准比较主观时，才可能引入 LLM-as-judge。

比如：

- 回答是否足够有帮助。
- 回答语气是否合适。
- 摘要是否完整。
- 两个版本哪个更好。

所以你要记住：

eval 先是评测思想，不是某个模型打分功能。

### 8. 测试和 eval 的共同点

测试和 eval 不是完全无关。

它们有共同点。

共同点一：

都需要输入样本。

测试需要测试输入。

eval 需要评测样本。

共同点二：

都需要预期。

测试需要 expected result。

eval 需要 expected output、reference output、rubric 或评价标准。

共同点三：

都需要可重复。

测试要能反复跑。

eval 也要能反复跑。

共同点四：

都服务于质量。

测试保证代码不坏。

eval 衡量效果好坏。

共同点五：

都可以进入自动化流程。

测试可以进 CI。

eval 也可以进 CI 或定期运行。

所以它们不是敌人。

更准确的关系是：

测试是工程质量的一部分。

eval 是 AI 效果质量的一部分。

在 AI 项目里，两者都要有。

### 9. 测试和 eval 的关键区别

虽然有共同点，但区别更重要。

测试通常更关心：

代码是否正确。

eval 通常更关心：

系统表现是否足够好。

测试更像门禁。

不通过就不能合并。

eval 更像体检报告。

它告诉你哪里强、哪里弱、是否比上一版更好。

测试的结果通常是：

- pass。
- fail。

eval 的结果可以是：

- pass/fail。
- 分数。
- 准确率。
- 召回率。
- 平均得分。
- 等级。
- 对比胜率。
- bad case 列表。
- 指标趋势。

测试通常要求强确定性。

eval 可以容纳一定模糊性。

比如：

测试：

```text
工具参数必须包含 order_id。
```

eval：

```text
回答是否足够解决用户问题。
```

测试不适合大量主观判断。

eval 更适合衡量主观质量和整体行为。

### 10. 为什么 eval 的结果经常不是绝对真理

测试失败通常很明确。

比如：

预期 200，实际 500。

这就是错。

但 eval 结果不一定这么简单。

比如你让 LLM-as-judge 给一个回答打 4 分。

这个 4 分不一定是绝对真理。

它受很多因素影响：

- 评分标准是否清楚。
- judge 模型是否稳定。
- 样本是否有代表性。
- 参考答案是否写得好。
- 业务标准是否明确。
- 是否需要人工复核。

所以 eval 结果要谨慎看。

不要把 eval 分数当成神圣真理。

正确做法是：

用 eval 找趋势、找差异、找坏例。

比如：

- 新 prompt 比旧 prompt 在字段提取上提升明显。
- 新模型回答更流畅，但无资料拒答变差。
- RAG top_k 调大后召回提高，但引用噪声也增加。
- 工单路线评测发现“投诉但缺少订单号”经常被误判成普通咨询。

这些结论比单个分数更重要。

### 11. 为什么测试不能覆盖 AI 质量

假设你写了一个 RAG 接口测试：

```python
assert response.status_code == 200
assert "answer" in response.json()
assert "citations" in response.json()
```

这个测试能证明什么？

它能证明：

- 接口没崩。
- 返回 JSON 结构对。
- 有 `answer` 字段。
- 有 `citations` 字段。

但它不能证明：

- answer 是否答对。
- citations 是否真的相关。
- 模型有没有编造。
- 回答是否完整。
- 用户是否能理解。
- 低相关检索是否被拒答。

所以只靠测试，你可能得到一个很危险的系统：

接口每次都 200。

JSON 结构每次都对。

但是回答质量很差。

这就是为什么 AI 应用必须补 eval。

### 12. 为什么 eval 不能替代测试

反过来也一样。

eval 也不能替代测试。

比如你跑了 100 条 Agent eval，平均分 92。

这能证明什么？

它能说明这些样本上表现不错。

但不能证明：

- `/health` 接口一定正常。
- Pydantic schema 一定不接受脏数据。
- API key 一定不会泄漏。
- Java 调用超时一定映射正确。
- 创建工单前一定要求用户确认。
- 幂等键冲突一定返回正确错误。
- 日志里一定带 trace_id。

这些确定性工程规则，必须靠测试守住。

如果用 eval 来判断“是否必须用户确认”，就不够严谨。

因为这是安全边界。

安全边界不是“模型大多数时候做对”就可以。

安全边界必须是：

每次都必须做对。

这就应该用测试。

### 13. 当前项目里 test 的典型对象

我们当前项目里适合 test 的东西很多。

比如 `projects/ai-service`：

1. Pydantic schema。

   请求字段缺失是否报错。

   类型错误是否报错。

   输出字段是否符合契约。

2. FastAPI router。

   路径是否存在。

   状态码是否正确。

   错误响应格式是否统一。

3. service 层。

   fake client 是否被正确调用。

   异常是否被映射成 AppException。

   超时是否变成指定错误码。

4. Tool Calling。

   工具名是否合法。

   参数是否通过 Pydantic 校验。

   未授权工具是否拒绝。

   写操作是否需要确认。

5. RAG。

   文档加载是否返回 Document。

   chunk 切分是否保留 metadata。

   filter 是否按权限过滤。

   score_threshold 是否过滤低分结果。

6. LangGraph。

   节点是否写入正确 State 字段。

   条件边是否按状态路由。

   interrupt 是否返回结构化 payload。

   checkpoint 是否能恢复。

这些都可以写成很明确的断言。

所以它们适合 test。

### 14. 当前项目里 eval 的典型对象

当前项目里适合 eval 的东西也很多。

比如：

1. 意图识别。

   用户说“订单一直不发货，我很着急”，应该是订单/工单相关，不应该当普通聊天。

2. RAG 回答质量。

   问“退款多久到账”，回答应该基于退款政策，不应该编造不存在的规则。

3. 无资料处理。

   知识库没有答案时，应该拒答或转人工/建工单，不应该硬编。

4. 字段提取。

   用户说“订单 A1001，手机 138xxxx，想催发货”，应该提取订单号、联系方式、问题类型和诉求。

5. 缺字段追问。

   用户只说“我要投诉”，Agent 应该追问必要信息，而不是直接创建工单。

6. 工具选择。

   查询订单状态时应该请求 `query_order`，创建工单前必须先确认。

7. prompt 版本对比。

   新 prompt 是否比旧 prompt 更少误判、字段提取更稳定。

8. 模型版本对比。

   换模型后是否在中文客服场景里更稳定。

这些问题很难只靠一两个单元测试覆盖。

它们需要一批代表性样本。

这就是 eval 的范围。

### 15. eval 的最小组成

一个最小 eval 通常需要四样东西：

1. 样本。

   也就是输入。

2. 预期。

   也就是你希望系统做到什么。

3. 运行目标。

   也就是你要评测哪个函数、接口、Agent 或流程。

4. 评判方式。

   也就是怎么判断 actual output 是否达标。

例如：

```json
{
  "id": "intent_001",
  "input": {
    "message": "我想知道退款多久到账"
  },
  "expected": {
    "intent": "policy_question"
  }
}
```

运行目标：

```text
classify_intent_node
```

评判方式：

```text
actual.intent == expected.intent
```

这就是一个很简单的 eval。

它不需要 LLM-as-judge。

它不需要复杂平台。

它先把样本、预期、目标和评判方式固定下来。

后面我们写评测脚本，就是把这个结构系统化。

### 16. eval 里的 evaluator 是什么

evaluator 可以理解成评判器。

它负责把 actual output 和标准进行比较，然后给结果。

比如：

输入：

```json
{
  "expected_intent": "policy_question",
  "actual_intent": "policy_question"
}
```

evaluator 输出：

```json
{
  "metric": "intent_match",
  "passed": true,
  "score": 1.0
}
```

再比如：

输入：

```json
{
  "expected_order_id": "A1001",
  "actual_order_id": "A1001"
}
```

evaluator 输出：

```json
{
  "metric": "order_id_match",
  "passed": true,
  "score": 1.0
}
```

再比如回答质量：

输入：

```json
{
  "question": "退款多久到账？",
  "answer": "退款一般 1-3 个工作日到账，具体以支付渠道为准。",
  "reference": "退款通常 1-3 个工作日到账。"
}
```

evaluator 可能输出：

```json
{
  "metric": "answer_correctness",
  "score": 0.9,
  "comment": "回答和参考规则基本一致。"
}
```

不同 evaluator 适合不同任务。

结构化字段适合代码 evaluator。

回答质量可能适合 LLM-as-judge 或人工 review。

安全边界适合硬规则 evaluator。

### 17. eval 里的 metric 是什么

metric 是指标。

指标是评测结果的名字和数值。

比如：

- `intent_accuracy`：意图识别准确率。
- `route_accuracy`：路线准确率。
- `field_exact_match`：字段完全匹配率。
- `order_id_match`：订单号匹配率。
- `citation_required_pass_rate`：引用要求通过率。
- `no_context_refusal_rate`：无资料拒答通过率。
- `tool_call_accuracy`：工具调用准确率。
- `unsafe_action_block_rate`：危险操作拦截率。

metric 的意义是：

让质量可以被观察、比较、追踪。

没有 metric，你只能说：

感觉比以前好。

有 metric，你可以说：

字段提取准确率从 78% 提升到 88%，但无资料拒答通过率从 100% 降到 92%，需要继续看 bad case。

这就更像工程表达。

### 18. eval 里的 bad case 为什么重要

eval 不是只为了得到一个分数。

更重要的是发现 bad case。

bad case 是失败样本。

比如：

```text
用户输入：我的订单昨天显示已发货，但物流一直没有动
预期 intent：order_query
实际 intent：ticket_request
```

这个失败很有价值。

它告诉你：

模型或规则把“查询状态”误判成“创建工单”了。

你可以继续分析：

- 是关键词“没有动”触发了投诉规则吗？
- 是 prompt 没讲清楚“先查询再建工单”吗？
- 是训练样本里类似表达太少吗？
- 是路由规则过于激进吗？

没有 bad case，你只看到一个数字。

有 bad case，你才知道怎么改。

所以高质量 eval 必须保存 bad case。

### 19. eval 和回归

回归评测是 eval 非常重要的用途。

假设你改了 prompt。

你希望新 prompt 解决一个问题：

让“我要投诉”这类输入更稳定地进入工单流程。

但是新 prompt 可能带来副作用：

把很多普通咨询也误判成工单。

这时就要跑旧的 eval cases。

如果旧样本大量失败，说明新 prompt 破坏了旧能力。

这叫回归问题。

测试里也有回归测试。

eval 里也有回归评测。

区别是：

测试回归更关注代码契约有没有坏。

eval 回归更关注 AI 行为有没有变差。

### 20. eval 可以写在 pytest 里

这里容易混。

pytest 是工具。

eval 是方法。

你完全可以用 pytest 来运行 eval。

比如：

```python
@pytest.mark.parametrize("case", eval_cases)
def test_intent_eval(case):
    actual = classify_intent(case["input"]["message"])
    assert actual == case["expected"]["intent"]
```

这段代码形式上是 pytest。

但它做的事情是 eval：

用一批样本评测意图识别效果。

所以不要把“工具”和“思想”混在一起。

pytest 可以跑测试。

pytest 也可以跑一部分 eval。

LangSmith 可以管理 eval。

本地脚本也可以跑 eval。

OpenAI 或其他平台也可以提供 eval 能力。

但真正重要的是：

你是否设计了好样本、好预期、好 evaluator、好报告。

### 21. 什么情况下 eval 应该进 CI

CI 指的是持续集成流程。

简单理解：

代码提交后自动跑检查。

测试非常适合进 CI。

因为测试通常快、确定、失败就阻止合并。

eval 能不能进 CI？

可以，但要分层。

适合进 CI 的 eval：

- 小规模。
- 低成本。
- 稳定。
- 规则明确。
- 失败可以明确定位。

比如：

- 意图识别 20 条核心样本。
- 工单字段提取 20 条核心样本。
- 无资料拒答 10 条安全样本。
- 写操作确认边界 10 条样本。

不适合一开始进 CI 的 eval：

- 大规模真实模型调用。
- 成本很高。
- LLM-as-judge 波动很大。
- 需要人工判断。
- 运行很慢。

这些可以先做手动评测、每日评测或发布前评测。

所以 eval 自动化不是一步到位。

要先做小而稳的核心 eval。

### 22. eval、人工验收和线上监控的关系

质量保障不是只有一种方式。

AI 应用通常需要四层：

1. 测试。

   保证代码契约。

2. 离线 eval。

   上线前评估效果。

3. 人工验收。

   对关键场景和主观质量做人工检查。

4. 在线监控。

   上线后观察真实流量、错误、延迟、成本和质量问题。

这四层不是互相替代。

它们负责不同问题。

测试解决：

代码有没有坏。

离线 eval 解决：

这版 AI 效果是否达到上线标准。

人工验收解决：

自动指标看不出来的业务体验和主观质量。

在线监控解决：

真实用户场景里有没有新问题。

阶段 6 会把这些逐步补齐。

本节先把测试和 eval 的边界讲清楚。

## 二、本节主题系统讲解

### 1. 一句话区分 test 和 eval

最容易记的说法是：

test 关注“对不对”。

eval 关注“好不好”。

再准确一点：

test 关注确定性契约是否被满足。

eval 关注 AI 行为质量是否达到业务标准。

例如：

用户确认后才能创建工单。

这是安全契约。

它必须永远成立。

适合 test。

RAG 回答是否足够完整、是否基于知识库、是否能帮用户解决问题。

这是效果质量。

适合 eval。

### 2. 用表格对比

| 维度 | test | eval |
| --- | --- | --- |
| 中文叫法 | 测试 | 评测 |
| 主要目标 | 验证代码行为符合预期 | 衡量 AI 应用效果是否达标 |
| 典型问题 | 有没有按规则运行 | 表现好不好 |
| 结果形式 | pass/fail 为主 | pass/fail、分数、准确率、坏例、对比报告 |
| 适合对象 | 函数、API、schema、异常、权限、路由硬规则 | 意图识别、RAG 回答、字段提取、工具选择、整体 Agent 表现 |
| 判断标准 | 明确、确定、硬断言 | 可以明确，也可以带一定主观性 |
| 是否可进 CI | 很适合 | 核心小样本适合，大规模评测可发布前或定期跑 |
| 成本 | 通常低 | 可能涉及模型调用，成本更高 |
| 失败含义 | 代码契约被破坏 | 某类样本效果不达标或质量下降 |
| 主要产物 | 测试用例、测试报告 | 评测集、指标、实验报告、bad case 列表 |

这个表要记住。

以后别人问你：

“你这个 Agent 怎么保证质量？”

你不能只回答：

“我写了很多测试。”

更完整的回答应该是：

“我用测试保证接口、节点、工具权限和错误处理这些确定性契约；再用 eval 固定样本评测意图识别、RAG 回答、字段提取和工具选择效果，并保存 bad case 做回归。”

### 3. 当前阶段为什么先讲 eval 而不是直接写代码

如果直接写评测脚本，你可能会看到很多 JSON、循环、断言、报告输出。

但是如果脑子里没有 test/eval 的边界，很容易出现三个问题：

1. 把 eval 写成普通单元测试。

   只检查 status code 和字段存在，不检查 AI 效果。

2. 把测试写成模糊评测。

   安全边界不用硬断言，反而让模型或主观判断决定。

3. 只看 pass rate，不看 bad case。

   看到 90% 通过就满意，但不知道失败样本会不会是最重要的业务场景。

所以第 2 节必须先讲清概念。

概念清楚了，后面第 3 节设计 Agent 测试集才不会乱。

### 4. 当前项目的四层质量保障

我们当前项目可以分成四层质量保障。

第一层：代码契约测试。

例如：

- `ChatRequest` 字段校验。
- `TicketFields` 字段校验。
- 工具参数 Pydantic 校验。
- Java client 超时映射。
- API 错误格式统一。

第二层：流程路线测试。

例如：

- `policy_question` 进入 RAG 节点。
- 字段缺失进入追问节点。
- 字段完整进入确认节点。
- 确认通过后才创建工单。
- interrupt 能暂停和恢复。

第三层：离线 eval。

例如：

- 50 条用户输入，意图识别准确率是多少。
- 30 条退款/账号/物流问题，RAG 回答是否有依据。
- 40 条工单表达，字段提取是否稳定。
- 10 条安全样本，是否拒绝危险操作。

第四层：线上监控。

例如：

- 哪些真实问题经常无资料。
- 哪些节点失败率高。
- 哪些模型调用延迟高。
- 哪些 bad case 应该加入离线 eval。

阶段 5 已经主要完成第一层和第二层的一部分。

阶段 6 开始补第三层和第四层。

### 5. 例子：Pydantic schema 应该用 test

假设我们有一个工单字段模型。

它要求 `issue_type` 只能是固定枚举。

如果传入：

```json
{
  "issue_type": "随便乱写"
}
```

应该校验失败。

这适合 test。

因为判断非常明确：

非法枚举不应该通过。

不需要 eval。

不需要大模型。

不需要人工评分。

直接写测试：

```python
with pytest.raises(ValidationError):
    TicketFields.model_validate(payload)
```

这就是正确方式。

### 6. 例子：退款回答质量应该用 eval

用户问：

```text
退款多久到账？
```

系统回答：

```text
退款通常 1-3 个工作日到账，具体以支付渠道处理时间为准。
```

这个回答是否好？

要看知识库规则。

如果知识库确实写了 1-3 个工作日，这个回答不错。

如果知识库写的是 3-7 个工作日，这个回答就是错的。

如果知识库没有写到账时间，模型却编了一个时间，就是幻觉。

这里不是简单的字段存在检查。

你需要评测：

- 是否引用正确知识。
- 是否回答了用户问题。
- 是否没有编造。
- 是否没有遗漏重要限制。
- 是否在无资料时拒答。

这更适合 eval。

### 7. 例子：创建工单前确认应该用 test + eval

有些场景两者都需要。

比如创建工单前确认。

从安全契约上看：

没有用户确认，不能调用创建工单工具。

这必须用 test。

但是从 Agent 效果上看：

面对不同表达，Agent 是否能准确判断什么时候需要进入确认？

比如：

- “帮我查一下订单”不应该直接进入工单确认。
- “我要投诉商家不发货”应该进入工单流程。
- “退款规则是什么”应该优先 RAG 回答。
- “你直接帮我取消订单”如果是敏感写操作，应该要求确认或拒绝。

这些输入表达很多样。

这就适合 eval。

所以一个能力可能同时需要 test 和 eval。

test 守硬边界。

eval 看覆盖面和效果。

### 8. 例子：意图识别应该怎么分

意图识别有两类验证。

第一类：规则和代码是否可运行。

比如：

- 返回值是否在合法枚举里。
- 空输入是否兜底。
- unknown intent 是否进入安全分支。

这些适合 test。

第二类：真实用户表达是否分对。

比如：

```text
我买的东西一直不动，帮我看下
```

它到底是订单查询、物流问题，还是工单创建？

这需要根据业务标准判断。

你要准备很多样本。

然后统计准确率和 bad case。

这适合 eval。

### 9. 例子：RAG 检索质量应该怎么分

RAG 里也一样。

适合 test 的：

- loader 能否加载文件。
- splitter 能否切 chunk。
- metadata 是否保留。
- filter 是否按权限过滤。
- no result 是否返回空列表。
- score_threshold 是否生效。

适合 eval 的：

- 某个问题 top_k 是否召回正确 chunk。
- 检索结果是否覆盖答案所需信息。
- 回答是否忠于检索内容。
- 引用来源是否真的支持答案。
- 无资料时是否拒答。

之前阶段 4 我们已经做过 RAG 检索评测。

阶段 6 会把 Agent 的评测也系统化。

### 10. 例子：prompt 改动为什么必须 eval

prompt 改动最典型。

你改一句提示词，代码测试可能全部通过。

因为：

- API 还是 200。
- JSON 还是合法。
- 字段还是存在。
- 节点还是能跑完。

但是模型行为可能变了。

比如：

旧 prompt 比较保守：

无资料时会拒答。

新 prompt 更积极：

无资料时也尝试回答。

测试可能发现不了。

eval 可以发现：

无资料拒答样本通过率下降。

所以 prompt 改动一定需要 eval。

### 11. 例子：模型升级为什么必须 eval

模型升级也一样。

比如从模型 A 换成模型 B。

模型 B 可能：

- 回答更流畅。
- 字段提取更好。
- 工具调用更积极。
- 但是更容易编造。
- 或者更容易跳过确认。

如果只看几个手动问题，很难判断。

正确做法是：

用同一批 eval cases 跑模型 A 和模型 B。

然后比较：

- intent 准确率。
- route 准确率。
- field match。
- no context refusal。
- unsafe action block。
- bad case 类型。
- 延迟和成本。

这才是工程化模型升级。

### 12. eval 报告应该怎么读

一个 eval 报告不应该只看总分。

应该看这几层：

1. 总体通过率。

   先看大概质量。

2. 分任务指标。

   比如 intent、RAG、field extraction、tool call 分开看。

3. 分业务场景指标。

   比如退款、账号、安全、物流、投诉分开看。

4. bad case。

   看失败样本具体是什么。

5. 和上一版对比。

   看是提升、下降，还是转移了问题。

6. 是否达到上线门槛。

   某些指标可以低一点继续优化，某些安全指标必须 100%。

比如：

```text
总通过率：86%
intent_accuracy：92%
field_exact_match：80%
no_context_refusal：100%
unsafe_action_block：100%
RAG answer faithfulness：78%
```

这个报告告诉你：

安全边界还可以。

意图识别不错。

RAG 回答忠实度偏弱。

字段提取还需要继续优化。

这比一句“效果还行”有用得多。

### 13. test 和 eval 在工作中的表达

以后你面试或跟别人讲项目时，可以这样表达：

```text
这个项目里我把质量保障分成两层。

第一层是传统测试，用 pytest 覆盖 schema 校验、API 契约、工具权限、错误处理、LangGraph 节点路由和 fake 依赖，保证确定性代码行为不坏。

第二层是 AI eval，用固定样本评测意图识别、RAG 回答、字段提取、工具选择和 Agent 路线效果，保留 bad case 并做回归评测。这样 prompt、模型、检索参数或节点逻辑改动后，不只看接口是否能跑，还能看 AI 行为质量有没有退化。
```

这个表达比：

```text
我写了测试。
```

专业很多。

也比：

```text
我用了 LangSmith。
```

更能说明你真的理解质量保障。

### 14. 本节和后续章节的连接

这节讲清楚了 test 和 eval 的边界。

接下来几节会按这个逻辑继续：

第 3 节：设计 Agent 测试集。

重点是：

哪些样本要放进去，expected output 怎么写。

第 4 节：意图识别评测。

重点是：

怎么用固定样本评估 intent。

第 5 节：工单字段提取评测。

重点是：

怎么评估 `order_id`、`issue_type`、`contact`、`description` 等字段。

第 6 节：Agent 路由评测。

重点是：

怎么判断流程路线是否合理。

第 7 节：RAG + Agent 组合评测。

重点是：

RAG 回答和 Agent 决策混在一起时怎么评。

第 8-12 节：

开始把评测脚本、报告、坏例分析、回归评测和 evaluator 类型落下来。

所以本节是后面 10 节的概念底座。

## 三、和当前项目的关系

### 1. 阶段 5 已经有很多 test

阶段 5 做 LangGraph 智能工单 Agent 时，我们已经写了不少测试。

这些测试主要保证：

- 图能运行。
- 节点能写入 State。
- 条件边能路由。
- fake RAG 能替换真实依赖。
- fake Java client 能模拟调用。
- interrupt 能暂停和恢复。
- 创建工单失败能 fallback。
- 日志能带 trace_id。

这很重要。

如果没有这些测试，阶段 6 很难继续往生产化推进。

因为每次加 eval、真实 LLM、真实工具、checkpoint 持久化时，都可能破坏旧逻辑。

所以我们不是抛弃测试。

我们是在测试基础上继续加 eval。

### 2. 阶段 6 要补的是 AI 行为质量

阶段 5 的测试更关注：

流程有没有跑通。

阶段 6 要补的是：

流程跑通之后，质量到底怎么样。

比如：

当前 Agent 能进入 RAG 节点。

但它是否在正确的问题上进入 RAG？

当前 Agent 能提取字段。

但它面对 50 种真实中文表达时提取准不准？

当前 Agent 能调用创建工单。

但它是否总是在用户确认后才调用？

当前 RAG 能返回答案。

但答案是否忠于知识库？

这些就是 eval 要解决的问题。

### 3. 当前第 2 节不写代码的原因

这一节继续不写业务代码。

原因不是偷懒。

而是这节的核心学习目标是建立边界：

什么该 test。

什么该 eval。

如果边界不清楚，后面写代码会变成机械堆文件。

你会看到一堆 JSON 和 pytest，但不知道为什么这么设计。

等第 3 节开始设计 Agent 测试集，第 4 节开始做意图识别评测时，代码才真正有学习意义。

所以这节以笔记为主是合理的。

## 四、常见误区

### 误区 1：eval 就是测试

不准确。

eval 和测试有重叠，但目标不同。

测试更偏代码契约。

eval 更偏 AI 效果。

可以用 pytest 跑 eval，但不能因此说 eval 就是 pytest。

### 误区 2：有 eval 就不用测试

错误。

eval 不能替代确定性测试。

权限、安全、异常、schema、状态码、用户确认边界，这些必须用测试守住。

尤其是写操作安全，不能靠模型“多数情况下会做对”。

### 误区 3：有测试就不用 eval

错误。

测试可以证明代码没崩，但不能证明 AI 回答有帮助。

如果只写测试，可能接口都正常，但模型回答一直编。

### 误区 4：eval 一定要 LLM-as-judge

错误。

很多 eval 可以用规则完成。

比如：

- intent 是否等于预期。
- route 是否等于预期。
- 字段是否匹配。
- citations 是否为空。
- 是否调用了危险工具。

LLM-as-judge 只适合某些主观判断。

### 误区 5：eval 分数越高越好

不一定。

分数要结合样本看。

如果样本太简单，分数高没有意义。

如果安全样本失败，即使总分高也不能上线。

如果新版本总分高，但关键业务场景退化，也要谨慎。

### 误区 6：只看平均分

平均分会掩盖问题。

比如 100 条样本，99 条普通问答通过，1 条危险写操作失败。

平均分可能很高。

但危险写操作失败非常严重。

所以 eval 要看分项指标和 bad case。

### 误区 7：eval 样本越多越好

不是一开始越多越好。

初期应该先做少量高质量样本。

比如每类核心场景 5-10 条。

样本要有代表性。

样本太多但质量低，会增加维护成本，还可能误导评测结果。

### 误区 8：平台工具等于评测能力

错误。

LangSmith、OpenAI Evals、内部评测平台都只是工具。

真正的能力是：

- 能不能设计样本。
- 能不能写预期。
- 能不能选 evaluator。
- 能不能看 bad case。
- 能不能做回归。
- 能不能把评测结论转成改进动作。

## 五、本节练习

### 练习 1：用一句话解释 test

题目：

用自己的话解释什么是 test。

参考答案：

test 是用明确输入、明确操作和明确断言，验证某段代码或某个流程是否符合预期的方法。

### 练习 2：用一句话解释 eval

题目：

用自己的话解释什么是 eval。

参考答案：

eval 是用固定样本和明确评判标准，系统化衡量 AI 应用在某些任务上的表现是否达标。

### 练习 3：为什么 AI 项目只写 pytest 不够

题目：

请说明为什么一个 AI 项目只有 pytest 还不够。

参考答案：

pytest 可以保证接口、schema、异常、路由等确定性代码行为不坏，但很难直接衡量模型回答是否有帮助、RAG 是否忠于知识库、字段提取在多种真实表达中是否稳定、prompt 或模型升级后效果是否变好，所以还需要 eval。

### 练习 4：为什么 eval 不能替代 pytest

题目：

请说明为什么有了 eval 也不能不写 pytest。

参考答案：

eval 衡量 AI 效果，但不能替代确定性工程测试。API 状态码、字段校验、错误映射、权限边界、写操作确认、工具参数校验等规则必须用 pytest 这类测试做硬断言，不能靠评测分数或模型判断。

### 练习 5：判断下面场景该用 test 还是 eval

题目：

判断以下场景更适合 test 还是 eval，并说明原因。

1. 缺少 `order_id` 时，`QueryOrderArgs` 校验失败。
2. 用户问“退款多久到账”时，回答是否准确并基于知识库。
3. 创建工单前必须有用户确认。
4. 50 条真实用户表达里，意图识别准确率是多少。

参考答案：

1. 更适合 test。因为这是明确的参数校验规则。
2. 更适合 eval。因为这是回答质量和知识忠实度问题。
3. 必须用 test，同时也可以用 eval 覆盖多种表达。test 守安全硬边界，eval 看复杂输入下是否稳定。
4. 更适合 eval。因为这是用样本集衡量 AI 行为效果。

### 练习 6：解释“pytest 可以跑 eval”

题目：

为什么说 pytest 可以跑 eval，但 eval 不等于 pytest？

参考答案：

pytest 是运行测试和断言的工具，eval 是用固定样本和指标评测 AI 效果的方法。我们可以用 pytest 参数化运行 eval cases，但 eval 的关键在于样本、预期、evaluator、指标和报告，而不是某个具体运行工具。

### 练习 7：给当前项目举两个 test 例子和两个 eval 例子

题目：

结合智能工单 Agent，分别举两个适合 test 的例子和两个适合 eval 的例子。

参考答案：

适合 test 的例子：

1. 工具参数缺少 `order_id` 时必须校验失败。
2. 没有用户确认时不能调用创建工单工具。

适合 eval 的例子：

1. 评测 50 条用户输入的意图识别准确率。
2. 评测退款、账号、物流等问题的 RAG 回答是否忠于知识库。

### 练习 8：解释 bad case 的价值

题目：

为什么 eval 不能只看总分，还要看 bad case？

参考答案：

总分只能告诉我们整体表现，bad case 才能告诉我们具体哪里错、为什么错、该改 prompt、检索、路由还是字段提取逻辑。没有 bad case，就很难把评测结果转成实际改进动作。

## 六、自测题

### 自测 1：本节最核心的一句话是什么

题目：

用一句话总结本节。

参考答案：

测试保证确定性代码契约不坏，eval 衡量 AI 行为和整体效果是否达标，AI 应用两者都需要。

### 自测 2：test 的典型结构是什么

题目：

test 常见的四个步骤是什么？

参考答案：

Arrange、Act、Assert、Cleanup。也就是准备、执行、断言、清理。

### 自测 3：eval 的最小组成是什么

题目：

一个最小 eval 通常需要哪四样东西？

参考答案：

样本、预期、运行目标、评判方式。

### 自测 4：为什么 prompt 改动必须跑 eval

题目：

为什么 prompt 改动后，代码测试全通过也不能说明质量没问题？

参考答案：

因为 prompt 改动可能不影响接口、JSON 结构和代码流程，但会改变模型回答、拒答、字段提取、工具选择和路线决策等 AI 行为，所以必须用固定样本评测效果变化。

### 自测 5：什么内容必须用 test 守住

题目：

列出三个必须用 test 守住的内容。

参考答案：

示例：Pydantic schema 校验、API 状态码和错误格式、创建工单前必须用户确认、工具参数校验、未授权工具拒绝、超时错误映射。

### 自测 6：什么内容更适合 eval

题目：

列出三个更适合 eval 的内容。

参考答案：

示例：意图识别准确率、RAG 回答忠实度、字段提取准确率、工具选择准确率、无资料拒答质量、模型版本对比、prompt 版本对比。

### 自测 7：eval 分数为什么不是绝对真理

题目：

为什么不能把 eval 分数当成绝对真理？

参考答案：

因为 eval 分数受样本质量、评判标准、evaluator 设计、judge 模型稳定性、业务标准和人工复核质量影响。分数更适合用于趋势、对比和坏例分析，而不是单独作为绝对判断。

### 自测 8：如何向别人解释本项目的质量保障

题目：

用 1 分钟向别人解释当前智能工单 Agent 如何保证质量。

参考答案：

这个项目会把质量保障分成两层。第一层是 pytest 测试，用来覆盖 schema、API、工具权限、错误处理、LangGraph 节点路由、fake 依赖和用户确认边界，保证确定性工程逻辑不坏。第二层是 Agent eval，用固定样本评测意图识别、RAG 回答、字段提取、工具选择和整体路线效果，并保存 bad case 做回归。这样后续改 prompt、模型、检索参数或节点逻辑时，不只看代码能不能跑，还能看 AI 行为质量有没有退化。

## 七、本节小结

本节你要真正记住的是：

test 和 eval 都重要，但负责的问题不同。

test 负责确定性工程规则。

eval 负责 AI 行为质量。

test 更像门禁。

eval 更像体检报告。

test 失败，通常说明代码契约坏了。

eval 下降，通常说明模型、prompt、检索、路由或业务策略需要继续分析。

当前智能工单 Agent 后续要继续变强，不能只靠“能跑通”。

我们要做到：

- schema 能测。
- API 能测。
- 节点能测。
- 权限能测。
- 错误能测。
- 意图能评。
- RAG 能评。
- 字段能评。
- 路线能评。
- 工具选择能评。
- bad case 能保存。
- 改动后能回归。

这就是阶段 6 开始补生产化和评测的原因。
