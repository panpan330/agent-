# 阶段 4 第 1 节：RAG 是什么，为什么大模型需要知识库

> 本节结论：RAG 的核心不是“让模型变聪明”，而是“先从外部知识库找资料，再让模型基于资料回答”。它解决的是大模型不知道企业私有知识、知识可能过期、回答缺少出处、不能直接追溯来源的问题。企业知识库问答不能只靠 prompt，必须把文档、检索、引用、权限和评测做成工程链路。

## 生成笔记前的教学复核

本节是阶段 4 的第一节，只讲 RAG 的底层概念和学习地图，不急着写代码。

这一节必须讲清：

```text
1. RAG 是什么。
2. 为什么大模型需要外部知识库。
3. 普通聊天、prompt、微调、Tool Calling、RAG 分别解决什么问题。
4. 企业知识库问答为什么不能只靠模型记忆。
5. RAG 的完整流程是什么。
6. 文档、chunk、embedding、向量库、检索、生成、引用这些词分别是什么意思。
7. 当前阶段为什么先学 Qdrant，再补 Milvus。
8. 本节不写代码，但要建立后续 36 节的知识地图。
```

## 本节一句话定位

第 1 节要解决的是一个根问题：

```text
大模型已经能聊天了，为什么还要专门做 RAG？
```

如果这个问题没想清楚，后面学习 chunk、embedding、Qdrant、Milvus、rerank、引用来源都会变成机械记 API。

## 先给结论

RAG 是：

```text
Retrieval-Augmented Generation
检索增强生成
```

拆开看：

```text
Retrieval：检索，从知识库里找相关资料
Augmented：增强，把找到的资料补充给模型
Generation：生成，模型基于资料生成回答
```

所以 RAG 的一句话解释是：

```text
先检索资料，再让模型基于资料回答。
```

它不是：

```text
把所有文档塞进 prompt
让模型自己记住公司知识
训练一个新模型
让模型直接查数据库
让模型随便编一个答案
```

它是一个工程流程：

```text
用户问题
-> 检索相关文档片段
-> 把片段作为上下文交给模型
-> 模型基于上下文回答
-> 返回答案和引用来源
```

## 为什么大模型需要知识库

你可能会自然地想：

```text
大模型不是已经学过很多知识了吗？
为什么还要知识库？
```

原因至少有 6 个。

### 1. 大模型不知道你的企业私有知识

模型训练时不可能知道你公司内部的：

```text
售后政策
退款规则
产品手册
内部 FAQ
项目文档
接口说明
运营流程
历史工单
员工手册
权限规则
```

这些资料没有进入模型训练数据。

所以你问：

```text
我们公司 A1001 类订单超过 72 小时未发货应该怎么处理？
```

模型如果没有你的内部文档，只能凭通用经验回答。

通用经验可能听起来合理，但不一定符合你的业务规则。

### 2. 模型知识可能过期

模型训练完成后，它的内部知识不会自动更新。

但企业资料经常变：

```text
政策改了
产品版本改了
接口字段改了
活动规则改了
售后流程改了
组织架构改了
```

如果只靠模型内部知识，它可能回答旧规则。

RAG 的做法是：

```text
不要求模型记住所有最新知识
而是在回答前检索最新文档
```

### 3. 模型会幻觉

幻觉就是模型生成了看似合理但实际不存在或不准确的内容。

例如它可能说：

```text
根据公司规定，超过 48 小时未发货可以自动赔付 20 元。
```

但你的公司文档里可能根本没有这条规定。

RAG 不能彻底消灭幻觉，但它能显著降低幻觉：

```text
因为模型回答时有可参考的文档片段
```

并且我们可以要求：

```text
找不到依据就说不知道
回答必须带引用来源
```

### 4. 企业问答需要可追溯

真实业务里，用户或同事经常会问：

```text
你这个答案依据哪份文档？
这条规则在哪一节？
这个结论能不能追溯？
```

普通聊天只能给答案。

RAG 可以返回：

```text
答案
引用来源
文档标题
章节
片段内容
相似度分数
```

这对企业应用非常重要。

因为很多时候：

```text
答案本身重要
依据同样重要
```

### 5. 上下文窗口不是无限的

就算你有所有文档，也不能每次都把全部文档塞给模型。

原因是：

```text
上下文窗口有限
token 成本高
输入太长会稀释重点
模型不一定能从超长上下文里稳定找到关键内容
```

RAG 的思路是：

```text
先用检索系统筛出最相关的少量片段
再把这些片段交给模型
```

它不是“全量塞文档”，而是“先找相关资料”。

### 6. 企业知识有权限边界

企业文档不是所有人都能看。

比如：

```text
普通客服只能看公开售后规则
主管可以看投诉升级规则
财务可以看退款审批规则
研发可以看内部接口文档
```

RAG 系统必须支持：

```text
按用户权限过滤文档
按部门过滤文档
按文档等级过滤文档
按项目空间过滤文档
```

这也是为什么后面要学 metadata 和 payload filter。

## RAG 解决的真实问题

RAG 解决的不是“模型不会说话”。

阶段 2 的 `/chat` 已经能让模型说话。

RAG 解决的是：

```text
模型回答没有企业知识依据
模型不知道最新资料
模型回答不能追溯来源
模型容易编造内部政策
长文档不能全部塞进 prompt
不同用户应该看到不同知识
```

把它放在企业知识库场景里，就是：

```text
用户问问题
系统从企业文档里找相关内容
模型基于找到的内容回答
答案带出处
没有资料就明确说不知道
```

这就是企业知识库 RAG 的基本价值。

## RAG 和普通聊天的区别

普通聊天：

```text
用户问题
-> 模型
-> 回答
```

RAG：

```text
用户问题
-> 检索知识库
-> 得到相关文档片段
-> 用户问题 + 文档片段
-> 模型
-> 基于资料的回答 + 引用来源
```

对比一下：

| 维度 | 普通聊天 | RAG |
| --- | --- | --- |
| 知识来源 | 模型内部知识和当前 prompt | 外部知识库 + 当前 prompt |
| 是否适合企业私有文档 | 不适合 | 适合 |
| 是否容易追溯来源 | 难 | 可以设计引用来源 |
| 知识更新方式 | 换模型或重新训练 | 更新知识库文档 |
| 成本控制 | 可能需要塞很多上下文 | 先检索再喂少量片段 |
| 权限控制 | 很难只靠模型实现 | 可在检索阶段过滤 |

一句话：

```text
普通聊天靠模型已有知识回答。
RAG 先查资料，再让模型回答。
```

## RAG 和 prompt 的区别

prompt 是给模型的指令。

例如：

```text
请你根据公司制度回答问题。如果不知道，请说不知道。
```

这个 prompt 本身不包含公司制度。

除非你把制度内容也放进 prompt。

但如果制度很多，就会遇到：

```text
上下文太长
成本太高
更新麻烦
权限难控
引用难做
```

RAG 不是替代 prompt。

RAG 是让 prompt 拿到更可靠的上下文。

更准确的关系是：

```text
prompt 负责告诉模型怎么用资料回答
RAG 负责把相关资料找出来
```

例如 RAG 里的 prompt 可能是：

```text
你是企业知识库助手。
请只根据下面提供的资料回答。
如果资料不足，请说“当前资料不足，无法确定”。
回答后列出引用来源。

资料：
{retrieved_chunks}

用户问题：
{question}
```

这里的关键是：

```text
retrieved_chunks 不是手写死的
而是检索系统动态找出来的
```

## RAG 和微调的区别

微调是训练模型。

RAG 是检索资料。

这两个经常被混淆。

### 微调更适合什么

微调更适合：

```text
调整输出风格
学习固定格式
学习特定任务模式
提高某类任务稳定性
```

比如：

```text
统一客服语气
把回复格式固定成某种风格
让模型更擅长某类分类任务
```

### RAG 更适合什么

RAG 更适合：

```text
大量外部知识
经常更新的文档
需要引用来源
需要权限过滤
企业内部资料问答
```

比如：

```text
公司制度问答
产品文档问答
接口文档问答
售后政策问答
项目知识库问答
```

### 简单对比

| 维度 | RAG | 微调 |
| --- | --- | --- |
| 核心动作 | 检索资料 | 训练模型 |
| 知识更新 | 更新文档和索引 | 重新训练或继续训练 |
| 是否适合频繁变化知识 | 适合 | 不太适合 |
| 是否天然有引用来源 | 容易做 | 不容易 |
| 是否适合权限过滤 | 适合 | 不适合 |
| 成本 | 主要是 embedding、存储、检索、生成 | 训练成本和维护成本更高 |

对我们当前主线来说：

```text
先学 RAG
后面需要时再讨论微调
```

## RAG 和 Tool Calling 的区别

阶段 3 我们刚学完 Tool Calling。

现在一定要把它和 RAG 区分清楚。

Tool Calling 解决的是：

```text
模型需要调用外部能力
```

例如：

```text
查订单
查物流
创建工单
查询退款
```

RAG 解决的是：

```text
模型需要查外部知识
```

例如：

```text
查产品手册
查售后政策
查内部 FAQ
查项目文档
查接口说明
```

对比：

| 维度 | Tool Calling | RAG |
| --- | --- | --- |
| 外部对象 | 工具/API/业务系统 | 文档/知识库 |
| 常见结果 | 结构化业务数据或执行结果 | 相关文本片段 |
| 典型例子 | `query_order(order_id)` | 检索“退款政策”相关段落 |
| 后端重点 | 权限、参数校验、确认、幂等 | chunk、embedding、metadata、召回、引用 |
| 风险点 | 模型越权调用工具、写操作误执行 | 检索错资料、引用错来源、权限泄露 |

以后完整智能工单 Agent 可能同时使用两者：

```text
用户问：订单 A1001 三天没发货，我能不能投诉？

Tool Calling：
-> 查询订单状态和物流状态

RAG：
-> 查询售后政策和投诉规则

模型：
-> 结合订单数据和政策资料生成回答
```

这才是更接近真实企业 AI 应用的形态。

## RAG 的完整流程

RAG 通常分成两个阶段。

第一阶段是离线或后台阶段：

```text
文档入库阶段
```

第二阶段是用户提问时：

```text
检索问答阶段
```

### 1. 文档入库阶段

流程是：

```text
load
-> clean
-> split
-> embed
-> store
```

翻译成人话：

```text
读取文档
-> 清洗文本
-> 切成小块
-> 把每块文本变成向量
-> 存进向量数据库
```

例如：

```text
售后政策.md
-> 去掉无用空行和噪声
-> 切成多个 chunk
-> 每个 chunk 生成 embedding
-> 存进 Qdrant
```

### 2. 检索问答阶段

流程是：

```text
question
-> embed query
-> retrieve
-> build prompt
-> generate
-> cite sources
```

翻译成人话：

```text
用户提问
-> 把问题变成向量
-> 去向量库找相似 chunk
-> 把 chunk 和问题一起放进 prompt
-> 模型基于资料回答
-> 返回引用来源
```

完整链路是：

```text
用户问题
-> 问题 embedding
-> Qdrant/Milvus 搜索相似向量
-> 拿到 top_k 文档片段
-> 过滤低相关片段
-> 组装 RAG prompt
-> 调用 LLM
-> 生成答案
-> 返回答案 + sources
```

## RAG 里的核心概念

### 1. Document

Document 就是一份原始资料。

例如：

```text
售后政策.md
产品说明.pdf
接口文档.docx
员工手册.txt
历史工单导出.csv
```

在 RAG 里，document 不一定只是一个文件。

它也可以是一条数据库记录、一篇网页、一条 FAQ。

### 2. Chunk

Chunk 是文档切出来的小片段。

为什么要切？

因为原始文档可能很长。

如果一整篇文档直接拿去做 embedding，粒度太粗。

比如一份 50 页手册，用户只问其中一个小规则。

如果整篇作为一个向量，检索会不够精确。

所以要切成：

```text
chunk 1：退款条件
chunk 2：退款时效
chunk 3：物流异常处理
chunk 4：投诉升级规则
```

后面检索时，可以只拿最相关的片段。

### 3. Metadata

Metadata 是 chunk 的附加信息。

例如：

```json
{
  "source": "售后政策.md",
  "title": "售后政策",
  "section": "物流异常处理",
  "doc_type": "policy",
  "department": "customer_service",
  "access_level": "internal",
  "updated_at": "2026-07-14"
}
```

metadata 的作用非常大：

```text
显示引用来源
按权限过滤
按文档类型过滤
按更新时间过滤
按部门过滤
排查检索结果
```

### 4. Embedding

Embedding 是把文本变成一组数字。

例如一句话：

```text
订单超过 72 小时未发货可以升级投诉。
```

会被 embedding 模型变成类似：

```text
[0.012, -0.231, 0.887, ...]
```

真实向量会有很多维。

我们不靠肉眼理解每个数字是什么意思。

我们只需要知道：

```text
语义相近的文本，向量距离更近。
语义不同的文本，向量距离更远。
```

所以用户问：

```text
订单三天没发货怎么办？
```

它的向量应该更接近：

```text
订单超过 72 小时未发货可以升级投诉。
```

### 5. Vector Store / Vector Database

向量数据库就是专门存储和搜索向量的数据库。

它要能做：

```text
存向量
存 metadata
按向量相似度搜索
按 metadata 过滤
返回 top_k 结果
支持更新和删除
支持索引
```

阶段 4 会先用 Qdrant。

后面再讲 Milvus。

### 6. Retrieval

Retrieval 是检索。

它要回答：

```text
用户这个问题，最相关的文档片段有哪些？
```

常见参数包括：

```text
top_k：返回前几个结果
score_threshold：相似度低于多少就不要
filter：按 metadata 过滤
```

### 7. Generation

Generation 是生成回答。

但在 RAG 里，生成不是让模型自由发挥。

而是要求：

```text
只根据检索到的资料回答
资料不足就说不知道
回答要带出处
不要编造引用
```

## 一个完整例子

假设知识库里有一段文档：

```text
标题：售后政策
章节：物流异常处理
内容：订单超过 72 小时仍未发货，客服应先查询订单状态和仓库处理记录；
如确认未出库，可创建投诉工单并标记为 high urgency。
```

用户问：

```text
订单三天没发货，我应该怎么处理？
```

RAG 系统会：

```text
1. 把用户问题转成向量。
2. 在向量库里找到“物流异常处理”这个 chunk。
3. 把该 chunk 放进 prompt。
4. 让模型基于这个 chunk 回答。
5. 返回答案和来源。
```

可能回答：

```text
根据《售后政策》中的“物流异常处理”章节，订单超过 72 小时仍未发货时，
客服应先查询订单状态和仓库处理记录；如果确认未出库，可以创建投诉工单，
并将紧急程度标记为 high。

来源：
- 售后政策.md / 物流异常处理
```

这个回答比普通聊天更可靠。

因为它有资料依据。

## 企业知识库 RAG 的关键价值

企业知识库 RAG 不是玩具功能。

它的价值在于：

```text
把企业内部知识变成可检索、可引用、可权限控制、可评测的问答系统。
```

它可以用于：

```text
客服知识库
售后政策问答
产品手册问答
研发文档问答
内部制度问答
项目交接问答
接口文档问答
历史工单辅助
```

对你当前学习路线来说，RAG 也非常关键。

因为后面的智能工单 Agent 需要同时知道：

```text
业务数据：订单、物流、退款、工单
企业知识：政策、规则、FAQ、历史处理方式
```

业务数据靠阶段 3 的 Tool Calling。

企业知识靠阶段 4 的 RAG。

## 为什么先学 Qdrant，再学 Milvus

阶段 4 会先用 Qdrant。

不是因为 Milvus 不重要。

而是因为学习顺序要控制复杂度。

### 先用 Qdrant 的原因

Qdrant 适合先把 RAG 主线跑通：

```text
collection
point
vector
payload
search
filter
```

这些概念和 RAG 主线贴得很近。

先用它可以把注意力放在：

```text
RAG 流程
chunk 设计
metadata 设计
检索结果质量
引用来源
测试评估
```

### 后学 Milvus 的原因

Milvus 也要讲。

但更适合放在后半段：

```text
Milvus 是什么
collection/schema/field/entity/index
Milvus Standalone
向量检索
scalar filter
索引基础
Qdrant vs Milvus
```

等你已经理解向量库通用概念后，再学 Milvus，会更容易对比：

```text
概念哪里相同
API 哪里不同
部署复杂度哪里不同
适合场景哪里不同
```

## 阶段 4 的完整学习地图

阶段 4 暂定 36 节。

可以分成 4 组。

| 分组 | 小节 | 主题 |

## 常见误区

### 误区 1：RAG 就是把文档塞进 prompt

不对。

RAG 的关键是检索。

不是每次塞全部文档。

正确做法是：

```text
先找相关片段
再把相关片段放进 prompt
```

### 误区 2：有了 RAG 就不会幻觉

不对。

RAG 能降低幻觉，但不能保证绝对不幻觉。

所以还要做：

```text
引用来源
资料不足时拒答
score_threshold
prompt 约束
测试评估
人工审核
```

### 误区 3：embedding 相似就一定答案正确

不对。

向量相似表示语义接近，但不等于答案一定正确。

可能出现：

```text
检索到相似但不适用的文档
检索到旧版本文档
检索到用户无权限看的文档
检索结果缺少关键上下文
```

所以后面要学 metadata、filter、score_threshold 和评测。

### 误区 4：RAG 可以替代数据库查询

不对。

RAG 适合查文档知识。

订单状态、库存数量、用户余额这类实时结构化数据，应该通过业务 API 或数据库查询工具获得。

也就是：

```text
文档知识 -> RAG
实时业务数据 -> Tool Calling / API
```

### 误区 5：向量数据库就是普通数据库

不完全对。

向量数据库除了存数据，还要支持：

```text
高维向量相似度搜索
向量索引
metadata/payload 过滤
top_k 检索
大规模近似搜索
```

它和 MySQL、PostgreSQL 的职责不同。

## 本节练习

### 练习 1：用自己的话解释 RAG

题目：

```text
不要使用“检索增强生成”这个词，尝试用普通话解释 RAG 是什么。
```

参考答案：

```text
RAG 就是用户提问后，系统先去企业文档里找和问题相关的资料，再把这些资料交给大模型，让大模型根据资料组织成自然语言答案，并告诉用户答案来自哪里。
```

### 练习 2：判断是否适合用 RAG

题目：

```text
下面哪些场景适合用 RAG？

1. 查询订单 A1001 当前物流状态
2. 回答公司售后政策
3. 回答产品安装手册里的步骤
4. 创建一张投诉工单
5. 查询内部接口文档字段含义
6. 判断退款申请是否符合公司规则
```

参考答案：

```text
适合 RAG：
2. 回答公司售后政策
3. 回答产品安装手册里的步骤
5. 查询内部接口文档字段含义
6. 判断退款申请是否符合公司规则中的“规则依据”部分

不主要靠 RAG：
1. 查询订单 A1001 当前物流状态，应该用 Tool Calling 调业务 API。
4. 创建一张投诉工单，应该用 Tool Calling + 用户确认 + 后端写操作。

第 6 条通常会组合使用：
RAG 查规则，Tool Calling 查订单/退款实际数据。
```

### 练习 3：区分 RAG 和 Tool Calling

题目：

```text
用户问：“订单 A1001 三天没发货，按公司规定能不能投诉？”

请说明哪些部分适合 Tool Calling，哪些部分适合 RAG。
```

参考答案：

```text
Tool Calling：
查询订单 A1001 的状态、发货时间、物流状态、是否已出库。

RAG：
检索公司售后政策、物流异常处理规则、投诉工单创建规则。

最终回答：
模型结合订单真实数据和政策文档，告诉用户是否符合投诉条件，并引用政策来源。
```

### 练习 4：为什么不能只靠 prompt

题目：

```text
如果 prompt 写“你必须根据公司政策回答”，但没有提供公司政策内容，会发生什么问题？
```

参考答案：

```text
模型没有真实政策内容，只能根据通用知识或猜测回答。它可能编造不存在的规则，也不能给出真实引用来源。prompt 可以约束回答方式，但不能凭空提供企业私有知识。
```

### 练习 5：设计 metadata

题目：

```text
如果有一份文档《售后政策.md》，你会给它切出来的 chunk 设计哪些 metadata？
```

参考答案：

```text
可以设计：

source: 售后政策.md
title: 售后政策
section: 具体章节名
doc_type: policy
department: customer_service
access_level: internal
version: 版本号
updated_at: 更新时间
chunk_index: chunk 序号

这些字段后面可以用于引用来源、权限过滤、版本排查和检索调试。
```

## 自测题

### 自测 1：RAG 的三个英文单词分别是什么意思？

参考答案：

```text
Retrieval：检索
Augmented：增强
Generation：生成

连起来就是：检索增强生成。
```

### 自测 2：RAG 为什么适合企业知识库？

参考答案：

```text
因为企业知识通常是私有的、经常更新的、需要权限控制的、需要引用来源的。RAG 可以从企业文档中检索相关片段，再让模型基于片段回答，比只靠模型内部知识更可靠。
```

### 自测 3：RAG 能不能替代 Tool Calling？

参考答案：

```text
不能。RAG 主要查文档知识，Tool Calling 主要调用业务 API 或工具。查询政策适合 RAG，查询订单实时状态适合 Tool Calling。真实系统经常需要两者结合。
```

### 自测 4：chunk 是什么，为什么要有 chunk？

参考答案：

```text
chunk 是从文档里切出来的小文本片段。因为原始文档可能很长，直接整篇检索粒度太粗、成本高、效果差。切成 chunk 后，系统可以检索到更精确的相关片段。
```

### 自测 5：embedding 是什么？

参考答案：

```text
embedding 是把文本转换成数字向量。语义相近的文本，向量距离通常更近。RAG 用 embedding 把文档片段和用户问题放到同一个向量空间里比较相似度。
```

### 自测 6：为什么 RAG 回答要带引用来源？

参考答案：

```text
因为企业应用需要可追溯。引用来源能让用户知道答案依据哪份文档、哪一节内容，也方便排查错误、更新资料和做质量评估。
```

### 自测 7：为什么阶段 4 先讲 Qdrant，再讲 Milvus？

参考答案：

```text
先用 Qdrant 是为了快速跑通 RAG 主线，把注意力放在文档、chunk、embedding、检索和引用来源上。等理解向量数据库通用概念后，再学 Milvus，可以更清楚地比较两种向量库的概念、部署、索引和适用场景。
```

## 本节总结

这一节你要记住一条主线：

```text
大模型不是企业知识库。
```

所以企业知识库问答不能只靠模型内部知识。

RAG 的核心流程是：

```text
先检索资料
再基于资料回答
最后给出引用来源
```

阶段 3 学的是：

```text
AI 怎么安全调用业务工具
```

阶段 4 学的是：

```text
AI 怎么可靠使用企业知识
```

未来智能工单 Agent 会把两者结合：

```text
Tool Calling 查业务数据
RAG 查企业规则和文档
LangGraph 编排多步骤流程
```
