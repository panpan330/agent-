## 一句话定位

当前学习方向是 **Java 后端转 AI 应用工程**。

不优先走纯算法、模型训练或只做提示词，而是把已有 Java 后端能力和 Python AI 生态结合起来，做能落地、可上线、可维护的 AI 应用。

## 学习协作主旨

后续学习必须始终按“从零基础教学”的方式推进。

核心要求：

- 不默认学习者已经懂某个概念。
- 每个新知识点先讲清楚“它是什么、为什么需要它、解决什么问题”。
- 再讲“它和已学知识的关系”，尽量用 Java 后端经验做类比，但不能因为会 Java 就跳过 Python/AI 基础。
- 然后给出最小可运行例子，通过代码验证理解。
- 最后补充常见错误、适用场景、边界和复习问题。
- 每节练习和自测问题都要在文档中提供参考答案，方便复盘。
- 对于重要知识点，需要提供可靠资料、官方文档、课程或视频方向，帮助系统学习。
- 学习目标不是只会照抄命令，而是能向别人解释清楚原理、用途、实践方式和常见坑。

教学节奏：

```text
概念解释 -> 为什么需要 -> 最小例子 -> 动手练习 -> 结果讲解 -> 常见问题 -> 自测问题 -> 笔记沉淀
```

判断是否学会：

```text
能自己写出来
能自己改出来
能说清楚为什么这么做
能解释出错时该怎么排查
能把它放进真实项目里使用
```

## 后续每节教学细则

在上面的主旨基础上，后续每节默认继续遵守这些细则：

1. 如果本节有新增、修改或删除业务代码，必须把这些代码讲得更细。测试代码不用逐行展开，但业务代码、配置、服务层、工具层、路由层、数据模型、错误处理和调用链路要讲清楚，让学习者真正知道每段代码为什么存在、解决什么问题、以后怎么改。
2. 尽量少重复以前已经讲过且几乎完全一样的内容。只有当旧知识在本节有新的使用方式、新的边界、新的风险或新的工程意义时，才重新提起并补充。
3. 每节内容要尽量丰富，不能只满足“代码能跑”。笔记要帮助学习者扩充知识储备，理解基础知识、工程背景、真实项目场景、常见误区、后续演进方向和可以向别人讲清楚的表达方式。
4. 测试部分不需要像业务代码一样讲得特别细。只讲重要测试：它验证了什么风险、为什么需要这个测试、失败时说明哪里可能有问题。重点仍然放在本节基础知识、扩充知识和增删改业务代码讲解上。
5. 笔记格式不必死板。最终目标是让学习者通过笔记完整学会对应知识点，未来开发、工作或向别人讲解时都能说得清楚、用得出来。
6. 如果需要检查项目里是否有中文乱码，看到 PowerShell 输出乱码时，应该优先怀疑是 PowerShell 输出编码把 UTF-8 中文显示错了。先用字节、编辑器、测试结果或其他方式确认文件真实内容，再决定是否修复，避免误判后做无意义的大范围改动。

## 背景判断

截至旧线程讨论时，结论是：

- Java 仍然适合企业后端、权限、数据库、业务流程、稳定接口。
- Python 是 AI 应用层、数据处理、模型 API、RAG、评测脚本的主力语言。
- AI 岗位机会真实存在，但初级竞争加剧，单纯“会调用模型 API”不够。
- 更现实的路线是 AI 应用工程：RAG、智能客服、Agent、知识库、工作流自动化、模型 API 接入、评测和工程化。

## 路线边界

优先路线：

```text
Java 后端
  负责登录、权限、业务 API、数据库、工单、订单、报表等稳定系统

Python AI 服务
  负责 LLM API、RAG、LangChain、LangGraph、工具编排、评测和 AI 流程

向量库/数据层
  使用 Qdrant、PostgreSQL + pgvector 等

工程化
  Docker、日志、追踪、权限、安全、成本统计、评测、部署文档
```

不作为当前主线：

- Spring AI：旧线程中明确因为本地/目标环境不流行，暂不作为主线。
- 纯算法岗：暂不优先学习模型训练、深度学习研究、GPU 训练。
- 全自动 Agent：先做可控流程，再逐步增加智能决策。

## 学习优先级

```text
RAG > Tool Calling > LangGraph > Eval/工程化 > Agent 花活
```

原因：

- RAG 是 AI 应用工程最常见、最能落地的能力。
- Tool Calling 能复用 Java 后端业务接口。
- LangGraph 适合有状态、有分支、可中断、可恢复、可审批的业务流程。
- Eval、日志、追踪、安全和部署决定项目能不能从 demo 变成真实服务。

### 第 2 周：LLM API 基础

目标：理解大模型应用不是只发一句 prompt。

重点：

- system prompt / user prompt
- temperature / max_tokens / top_p
- streaming
- structured output
- tool calling 基础

练习：

验收：

### 第 3 周：LangChain 核心组件

目标：会用 LangChain 组织模型、Prompt、工具和结构化输出。

重点：

- ChatModel、messages、prompt template

验收：

- 至少 3 个 tool
- 每次 tool 调用有日志
- 工具入参有 Pydantic 校验

### 第 4 周：Java 业务服务接入

目标：把 Java 后端能力接进 AI 系统。

Java 服务接口：


### 第 5 周：RAG 入门，文档入库

目标：完成知识库的数据处理链路。


重点：

- Markdown / txt / PDF / docx 解析

- metadata 设计

chunk metadata 至少包含：

```text

content

```

验收：

- 上传 PDF 后能切成 chunks

### 第 6 周：RAG 查询链路

目标：完成企业知识库问答 v1。

流程：

```text
用户提问
  -> 问题向量化
  -> 向量库 top_k 检索
  -> 拼接 context
  -> LLM 基于 context 回答
  -> 返回 answer + citations
```

必须实现：

- top_k 参数
- score_threshold
- 引用来源
- 无资料时拒答
- 检索日志

验收：

- 回答必须带引用
- 引用能定位到文档和 chunk
- 没有检索结果时不能胡编
- 至少准备 20 个测试问题

目标：让 RAG 从 demo 变成可用系统。

验收：

重点：

## 深度教学标准（长期有效）

学习笔记和课堂讲解不能只停留在“做了什么、代码如何运行”的工程实现说明；最终产物应当能用于长期复习、面试讲解和工作实践，帮助学习者真正学会而非复制实现。

每节必须遵守以下标准：

1. 新概念从基础讲起：它是什么、为什么需要、解决什么问题、没有它会有什么后果。
2. 每个关键概念同时提供人话解释、工程术语和当前项目中的具体例子。
3. 明确比较旧知识与新知识：本节比上一节多了什么、为什么现在学习、如何组合使用、后续会如何演进。
4. 如果涉及新增或修改业务代码，不能只列职责。对关键文件、类和函数必须说明：为什么新增或修改、输入是什么、返回什么、内部执行步骤、关键分支原因、异常处理，以及与前后模块的连接关系。
5. service、router、schema、tool、client 等核心代码必须沿调用链拆开讲；核心函数不能用一句话带过。
6. 每节补充真实项目视角，包括与本节有关的日志、trace_id、权限、成本、超时、重试、安全、审计、测试和部署关注点。
7. 每节补充常见误区和可执行的排查方法，不只描述正确流程。
8. 测试部分不需要逐行讲，但必须说明每类关键测试防止什么问题、失败时优先怀疑哪里。
9. 内容应当丰富，不因为节省篇幅而压缩关键解释；但所有扩展必须围绕本节主线，不能提前塞入后续章节内容。
10. 如果完成后发现笔记偏薄，主动增加“扩展讲解”，再结束本节，不等待学习者提醒。

每节笔记至少包含：

```text
练习和参考答案
自测题和参考答案

```

## 基础知识铺垫规则（长期有效）

每节笔记必须有明显独立的“基础知识铺垫”部分。它不能只围绕本节代码解释，也必须讲清支撑这些代码能够成立的底层概念。

总原则是：

```text
先建立知识体系和概念模型
-> 再用项目代码验证、观察和实践
-> 最后形成能迁移到工作、面试和新项目的理解
```

代码服务于学习，不是学习只服务于让项目跑通。项目能运行只是验证点之一，不能替代对原理、设计原因、边界、错误路径和真实工程取舍的理解。

### 代码讲解取舍规则

后续每节不再默认“新增了代码就必须大篇幅逐段讲代码”。代码是辅助学习和验证理解的材料，不是课堂本身。讲解重心应按下面顺序分配：

```text
基础概念和知识体系
-> 为什么需要这个设计
-> 真实项目里解决什么问题
-> 当前项目的关键调用链
-> 对理解有帮助的核心代码
-> 常见错误、排查方法和练习
```

代码讲解只在以下情况详细展开：

1. 这段代码承载了本节核心概念，不讲代码就无法真正理解本节；
2. 这段代码体现了重要设计取舍，例如权限边界、幂等、确认机制、错误映射、跨服务调用、数据校验；
3. 这段代码以后很可能需要学习者独立修改、扩展或排查；
4. 这段代码连接了多个模块，是理解完整调用链的关键节点；
5. 这段代码容易被误用，不讲清楚会造成错误理解或真实项目风险。

如果某段代码只是样板代码、重复旧模式、简单路由注册、普通字段声明、测试辅助或对本节学习帮助不大，可以只说明它的职责和位置，不做大篇幅讲解。测试代码保持“讲关键风险和验证目标”为主，不逐行展开。

每节笔记应避免把篇幅堆在代码清单上，而忽略基础知识、设计原因、真实工程场景、错误路径和排查方法。判断标准是：这段代码讲解是否真的帮助学习者建立可迁移的理解；如果不能，就压缩它。

### 学完后必须形成清晰收获

后续每节不能只堆基础知识、代码解释和测试结果。学习者学完后必须能马上回答一个核心问题：

```text
我这一节到底学会了什么？
```

如果学习者看完笔记后只能说“我看了很多文件、很多接口、很多概念”，但说不清本节新增了什么能力、解决了什么问题、和上一节有什么不同，就说明教学结构仍然不合格。

为避免这种情况，每节笔记必须显式给出以下内容：

1. 本节一句话定位：用一句话说明本节在整条路线里的位置；
2. 本节解决的真实问题：先讲“没有它会卡在哪里”，再讲实现；
3. 本节新增能力：学完后学习者能多做什么；
4. 和上一节的区别：上一节做到哪里，本节往前推进了哪一步；
5. 最小心智模型：用 3-6 行流程图或要点总结本节核心链路；
6. 不看代码复述版：用自然语言讲一遍本节，不依赖文件名和函数名；
7. 学会的判断标准：列出 3-5 个能自测的能力项。

代码学习结束后，还必须有一个“回到学习目标”的收束段，把刚才讲过的代码重新映射回本节目标。不能讲完代码就直接进入测试或结束，否则学习者容易只记住实现细节，忘记本节真正要建立的知识结构。

推荐每节结尾增加类似结构：

```text
本节你真正学会的是：
- 以前只能做到什么；
- 现在能做到什么；
- 这个能力在真实项目里解决什么问题；
- 如果让你向别人解释，你应该怎么说；
- 如果让你自己再做一遍，你应该先抓住哪条主线。
```

### 借鉴阶段二笔记的主题讲解结构

阶段二部分笔记的效果更好，原因是它们不是直接从“基础知识铺垫”跳到“代码实现”，而是在中间加入了大量围绕本节主题本身的系统讲解。后续笔记必须借鉴这种结构。

默认结构调整为：

```text
本节一句话定位
-> 本节解决的真实问题
-> 基础知识铺垫
-> 本节主题系统讲解
-> 最小心智模型
-> 当前项目如何落地
-> 关键代码讲解
-> 常见误区和排查
-> 练习与答案
-> 自测与答案
-> 本节真正学会了什么
```

其中“本节主题系统讲解”是强制内容，不能省略。它的作用是让学习者在看代码前先真正理解这节课本身，而不是只知道项目里新增了哪些文件。

这一部分应重点讲：

1. 本节主题完整是什么，不急着进入代码；
2. 它在真实项目里通常怎么工作；
3. 它和上一节、下一节的关系；
4. 常见设计方式有哪些，各自取舍是什么；
5. 初学者容易误解哪里；
6. 当前项目为什么选择这种做法；
7. 学完后应该形成什么心智模型。

代码讲解必须放在主题讲解之后。也就是说，先建立“这件事是什么、为什么、怎么判断对错”的理解，再用当前项目代码验证和落地。不能让代码实现替代主题教学。

### 基础知识和主题讲解必须成为主要篇幅

后续每节笔记中，“基础知识铺垫”和“本节主题系统讲解”必须是主要学习内容。这里的要求不是机械追求字数，而是必须足够支撑学习者真正理解本节主题。

默认篇幅重心应接近：

```text
基础知识铺垫 + 本节主题系统讲解：主要篇幅
关键代码讲解：辅助篇幅
测试说明：简要但覆盖关键风险
```

“基础知识铺垫”负责补齐理解本节前必须知道的概念、术语、背景、工程常识、相关协议、常见业务场景和前置能力。复杂主题要多讲，不能因为后面有代码实现就压缩基础知识。

“本节主题系统讲解”负责把这一节课本身讲透，包括它完整解决什么问题、真实项目通常怎么做、有哪些方案和取舍、本项目为什么选择当前做法、初学者容易误解什么、和前后章节如何连接、学完后应该形成什么心智模型。

如果一节笔记读起来主要是在解释文件、类、函数和测试，而“基础知识铺垫”和“本节主题系统讲解”偏短，就视为不符合后续学习要求，应先补充这两部分再结束本节。

### 学习资料唯一验收准则

学习者的主要学习材料是本仓库 `notes/` 中由助手生成的笔记，因此笔记质量必须稳定，不能在不同小节间出现明显的“工程说明很完整、知识讲解却偏薄”的落差。

后续每节笔记的唯一验收准则是：学习者是否能依靠笔记建立足够的知识体系，在未来做到：

```text
能独立开发相近能力
能在工作中理解设计和排查问题
能向同事、面试官或他人讲清概念、链路和取舍
```

在交付前，助手必须把笔记当作正式学习资料进行自检，而不是把“代码、测试和接口能跑”当作完成条件。至少检查：

1. 基础知识铺垫是否足以让学习者不依赖默认前置知识理解本节；
2. 关键概念是否同时说明原理、设计原因、项目例子、失败路径和排查方法；
3. 核心代码是否沿调用链解释到足以让学习者自行修改；
4. 是否明确了本节与已学知识、后续演进及真实生产约束的关系；
5. 学习者能否据此完成开发、工作排查和口头讲述，而不只是复述 API 或复制代码。

如果其中任一项明显不足，先补充笔记，再宣布本节完成。

### 生成笔记前的强制复核

每次开始生成新的学习笔记前，助手必须先反思并列出本节笔记应满足的教学要求，再开始写正文。这个步骤用于保证笔记质量，不能省略或只在心里模糊判断。

至少复核：

```text
本节的真正学习目标和明确边界是什么？
学习者需要哪些基础知识才能理解本节？
哪些旧知识在本节进入了新场景，需要重新解释？
哪些业务代码、关键函数和调用链必须详细讲？
哪些失败路径、误区和排查方式必须覆盖？
哪些真实项目约束值得补充？
如何设计手动验证、重要测试、练习、自测和下一节衔接？
```

只有确认上述内容已被纳入笔记设计后，才开始生成笔记正文；笔记结束前仍需按“学习资料唯一验收准则”再次检查质量。

每个基础概念按以下顺序讲解：

```text
这个概念是什么
-> 它为什么出现
-> 它解决什么问题
-> 没有它会遇到什么麻烦
-> 它在真实开发中常见在哪里
-> 它在当前项目里对应哪段代码
```

具体要求：

1. 基础知识点数量不设固定下限或上限。应按本节主题、前置知识和学习者可能的理解缺口决定，直到能够充分覆盖本节、支撑学习者真正掌握为止；简单主题可以更少，复杂主题必须更多。
2. 每个重要基础概念尽量同时提供人话解释、工程术语、小例子和当前项目例子。
3. 本节如果用到了此前没有系统讲过的基础概念，必须补讲，不能默认学习者已经知道。
4. 已经学过的概念如果在本节进入了新场景，也必须从新场景重新解释其作用和设计原因。
5. 不只讲“怎么写”，还要讲“为什么这样设计”；不只讲成功路径，也要讲典型失败路径和排查方法。
6. 对会影响未来工作、面试表达或故障排查的概念，增加更充分的解释和真实项目视角。
7. 可以用 Java 做类比帮助理解，但不能因为学习者学过 Java 而省略 Python、FastAPI、AI 服务和模型调用中的基础知识。

每节结束前自检：

```text
本节基础知识是否足够？
学习者能否向别人解释这些概念？
学习者是否知道这些概念在当前代码哪里体现？
学习者是否知道概念相关错误发生时怎么排查？
```
